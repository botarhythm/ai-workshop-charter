orch × Ocean — 統合運用の仕組み

憲章ページの「工房」を実装レベルで開けて見せます。独立に生まれた2つの型 orch（fable_orchestra / akiratsukakoshi） と Ocean（私たちの型）。その共通の哲学とそれぞれの独自性、両者を束ねた統一エントリ /ocean の流れ、モデル選択、他Skills連携、そして「修正を資産に変える学びのループ」まで。

01収束した2つの型

orch と Ocean は別々に育ちました。それぞれの設計者が、同じ問い——「高い最上位モデルのトークンを、何に使うべきか」——から出発し、驚くほど近い答えに独立して到達しています。とりわけ、両者とも最上位モデル（Fable）をちょうど3つの高判断ポイントに絞る点で一致します。

共通の柱orch での呼び方Ocean での呼び方
最上位モデルは"3点"だけP1設計 / 迷い域裁定 / P1検収AC設計 / returned診断 / 抜き取り監査
量のある仕事は割安モデルへ実装・調査をサブエージェントへ委譲実装・調査を下位モデルへハンドオフ
状態はファイルに外部化STATUS.md / cost-log.mdKNOWLEDGE.md / ADR / tasks/
検品は省略しない検収3点セット検証ループ＋抜き取り監査
改訂は現場の事実からlessons/各テンプレの改訂履歴

この一致が意味すること

別々の設計者が同じ配置に行き着いた——これはこの哲学が本質的に正しいという独立検証にほかならない。だからこそ、片方を否定するのではなく束ねるのが自然だった。

02それぞれの独自性

同じ哲学でも力点が違います。優劣ではなく、異なる強み。束ねると互いの穴を埋め合うのはこのためです。

観点orch の流儀Ocean の流儀
配布・共有共有ハブ＋pull追従。大勢が同じ型を採用し更新を受け取り続けられる（プロダクト的）自プロジェクト群に最適化した自作の型
モデル選択リスク採点 P1–P4（金銭/PII/不可逆/依存/定型）。明快で教えやすい3軸（曖昧さ×検出コスト×影響半径）。仕事の性質そのものを測る
委譲の方式セッション内サブエージェント。自動ファンアウトとトークン計測に強い文書ハンドオフ。別セッション・別AI（Antigravity等）にも手渡せる
状態・決定STATUS＋cost-logでコスト較正に寄せるKNOWLEDGE＋ADRで決定の"なぜ"を残す
改善の回路lessons/ に教訓を蓄積修正を型そのものに刻む学びのループ（§03）

orch は「配って育てる」設計、Ocean は「渡して学ぶ」設計。どちらも同じ哲学の別々の到達点です。

03修正を資産に変える学びのループ

Ocean の背骨にある、最も特徴的な仕組みです。「ここはこう直して」というあなたの一言を、その場の修正で終わらせない。

Ocean の背骨

直しは"事実"として型に吸収され、版が上がる

現場でつまずいた事実（あなたの細かい修正指示を含む）は、まず該当タスクの修正よりも先に、型・受け入れ条件（AC）の書き方そのものに反映される。改訂の根拠は必ず「現場で困った事実」に限る——理屈だけの改訂はしない。だから次からは、誰が回しても最初からそのルールが効く。

現場で困る / あなたが直す→ 型・ACに反映→ 改訂履歴に理由を刻む→ 全セッションに恒久適用

実際に型へ吸収された"現場の事実"の例

  • 全体テストをACにする前にベースラインを実測せよ（既存の赤があると、正しい実装ほど差し戻される事故が起きた）。
  • 秘密情報スキャンはコードだけでなくmd/yaml等の文書ファイルもgrepせよ（仕様書の中の平文パスワードを検出した実例）。
  • 実機依存タスクはPCテストのgreenを実機の正常と信じるな／1変更ずつ実機確認（音声AIでウェイクワードを壊した実例）。
  • 自己修正が入った合格（attempts>0）は全件監査に格上げ（AC通過だが意図未達がすり抜けた実例）。

なぜ資産なのか

普通、レビューの指摘は一度直せば消える。この型では、指摘が手順書の一行として残る。修正のたびに工房の知恵が増え、同じ失敗を二度としない。あなたが費やした"直す労力"が、複利で効く資本に変わる。

04/ocean 実行の流れ

統一エントリ /ocean は、Ocean の型で回し、orch由来の計器を差し込みます。橙 = orch由来の計器

0
読み込み（コールドスタート）CLAUDE.md → KNOWLEDGE.md → 自分のタスクだけ

ハブのテンプレ（読み取り専用）と、プロジェクトの現在状態だけを読む。コンテキスト＝コスト。

1
発注 / 作業の切り出しレビュー=05 / 新規=06 / 単発=ハンドオフ文書

受け入れ条件つきの単位に分解する。

1b
規模の実測計器scope_size.py

触るスコープの推定トークンと帯（直読圏/ハイブリッド/蒸留必須）を計測。

2
モデル選択（3軸）指令塔

曖昧さ×検出コスト×影響半径の3問で、Fable / Opus / Sonnet を決める（§05）。

3
見積り＋承認ゲート計器指令塔 → ユーザー

入出力トークン見込みを提示。F2ではFable投入は必ず承認を待つ。

4
実行（ハンドオフ）下位モデル（別セッション or サブエージェント）

タスク＝1ファイル（AC＋実行報告）。Fableは3点（AC設計/診断/監査）だけ。

5
検証ループAC機械実行＋抜き取り監査

2敗で差し戻し。不可逆・初回・機械AC無し・attempts>0 は全件監査へ格上げ。

6
記録＋学びのループKNOWLEDGE.md / ADR / tasks

状態と決定を残す。見つかったACの穴は§03のループで型へ還元。トークン実測は agent_usage.py。

05モデル選択（Ocean 3軸に一本化）

統合にあたり、モデル選択は Ocean の3軸に統一しました。タスクの種別ではなく、仕事の性質で選びます。

  1. Q1. 受け入れ条件を「コマンド＋期待結果」で書き切れるか？ 書き切れない（判断そのものが成果物）→ Fable。
  2. Q2. 失敗はACで自動検出でき、影響は対象ファイル内に閉じるか？ はい → Sonnet／道中に自律判断が残る → Opus。
  3. Q3. 不可逆操作（本番・公開・課金・削除）を含むか？ 含む → 実行はそのまま、計画承認と事後確認は Fable を通す（全件監査対象）。

2つの流儀が地続きになった点

orch のリスク採点にあった「探索性（答えが未知）」への配慮は、Ocean の Q1「ACを書き切れるか＝判断が成果物か」にもともと包含されている。だから3軸に寄せても、orch の勘所は失われない。迷ったら1段安い方へ——エスカレーション経路が下振れを回収する。

06実プロジェクトでの動き

あるEC監査プロジェクトでの回り方を、機密数値を伏せて示します。1つのゴールが複数の作業単位に割れ、それぞれ別の担当で処理されます。

作業単位3軸の見立て担当
停止していた顧客向けメール送信の復旧不可逆（誤送信は取り消せない）＝Q3該当調査はSonnet→対策設計と送信対象の判断はFable→承認ゲート→全件監査
未設定の商品説明メタを一括生成ACで全て規定できる定型Sonnet が一括ドラフト→人が承認
売上×在庫の優先リスト生成既知データの集計・道中に判断少Opus/Sonnet が自走
原因不明の「特定流入だけ転換ゼロ」の調査ACを書き切れない（判断が成果物）Fable が直営で仮説→検証

同じゴールでも、単位ごとにコストと厳格さが最適化される。危険な1件だけ最上位を厚く、量のある作業は割安に流す。

07他のSkillsとの連携

/ocean は横断プロセス（メタ・スキル）です。ドメインを持つ他スキルを置き換えず、その上に「選択・委譲・検証」の型をかぶせます。指令塔が指揮者、他スキルが楽器という関係です。

作業の種類3軸の見立て実作業で呼ぶスキル
ブランド文面・営業コンテンツOpus/Sonnetbrand-voice（生成）＋ quality-assurance（検証）
課金・サブスク導入不可逆・課金 → Fable承認stripe-provision / stripe-payment-link-kit
DBスキーマ変更・移行不可逆 → Fable承認Supabase系ツール（監査でロールバック確認）
納品物（資料・表・PDF）定型docx / pptx / xlsx / pdf
実装の検証検証ループの道具として/verify · /code-review · /security-review
調査・リサーチ常に安いモデルへdeep-research / Explore
  1. 指揮とスキルは別。 /ocean が「どの単位を・誰に・どの厳格さで」を決め、実作業は各ドメインスキルが担う。
  2. 検品は必ず型側で締める。 ドメインスキルの出力も /verify 等を道具に独立検証する。
  3. プロジェクトのCLAUDE.mdが最優先。 既存の運用があれば、その内側で選択・委譲・検証としてだけ働く。

08導入手順

仲間が自分の環境に入れる場合の最小手順です（~ はホーム。Windowsは各自のパスに読み替え）。

# 1. 方法論の芯(ハブ)を用意し、スキルを共通置き場へ
mkdir -p ~/.claude/skills
cp -r <Fable_Ocean>/skills/ocean ~/.claude/skills/

# 2. SKILL.md 冒頭の「ハブのパス」を自分のテンプレ置き場に合わせる

# 3. 各プロジェクトで /ocean を起動 → KNOWLEDGE.md / tasks/ が用意される

各プロジェクトの CLAUDE.md に運用規約（F1/F2フェーズ・承認ゲート）を数行置けば、そのプロジェクト固有の前提が型より優先されます。改善は常に「現場で困った事実」を根拠に、型そのものへ還元してください（§03）。

出自と敬意

この統合型は、Ocean（私たちの型）を背骨に、盟友 akiratsukakoshi の orch（fable_orchestra）から計器（トークン実測・フェーズ規律）を受け取って成り立っています。同じ問いから独立に生まれた同志の設計に敬意を。
