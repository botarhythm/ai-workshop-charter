01ハブ＆スポーク構造

この仕組みは1個のスキルではなく、「読み取り専用の型（ハブ）」と「プロジェクトごとの状態（スポーク）」の分離で成り立ちます。ハブは全プロジェクト共通の思考法を保持し、スポークは各プロジェクト固有の進捗・決定・コストを保持します。

STATUS.md

CLAUDE.md
  • ハブは一度cloneして読み取り専用。git pull で作者の更新を受け取り続ける。書き込みは絶対にしない。
  • スキル（/orch）はユーザー共通のスキル置き場に置き、どのプロジェクトからでも起動できる。
  • スポークは各プロジェクト配下の docs/orchestration/。状態を外部化するので、セッションやモデルをまたいで再開できる。
  • 私たちの拡張は各プロジェクトの CLAUDE.md に置く。SKILL.md本体を触らないので、pullで更新が来ても壊れない。

設計上の勘所

「共有の知恵（ハブ）」と「個別の状態（スポーク）」を物理的に分けたことで、更新の追従とプロジェクト独立性が両立する。カスタムは常にCLAUDE.md側＝上書き層に置くのが鉄則。

02/orch 実行の流れ

タスクを渡すと、指令塔役のモデルが次の順で回します。橙 = 私たちが挿入した拡張

1
WP切り出し指令塔

タスクを「受け入れ条件つきで1回の委譲・検収で完結する単位（Work Package）」へ分解。各WPに目的・検証可能な合格条件・依存関係を付す。

1b
規模の実測拡張scope_size.py

触るサブシステムの推定トークンを1コマンドで計測し、直読圏/ハイブリッド/蒸留必須の帯を判定。読み込み分担と見積りの土台にする。

2
パターン判定指令塔（迷い域のみ上位モデル）

5問の採点シート（金銭+2 / 個人情報+2 / 不可逆+2 / 依存広い+1 / 定型−2）でP1〜P4を機械的に決める。

2b
蒸留耐性テスト → P0判定拡張指令塔

リスクとは別に「答えは既知の要約から導けるか／未知の発見が要るか」を自問。後者ならP0（天才直営）へ。

3
見積り提示指令塔 → ユーザー

WPごとにパターン・分担・入出力トークン見込みを表で提示。従量発生見込みや大型WLは着手承認を待つ。

4
実行（パターン別の分担）指令塔 + 実行部隊サブエージェント

設計は指令塔、実装は割安モデル、調査は必ず安いモデルへ。詳細は §04。

5
検収3点セット指令塔

(1) 合格条件を自分のツールで再実行 (2) 差分を目視 (3) 成果物を実読し「悪用されたら？失敗したら？」を自問。P1〜P3では省略禁止。

6
実績記録指令塔 → スポーク

入出力トークンの実測を cost-log.md に、現在地を STATUS.md に追記。見積り精度が回を追って上がる。

03パターン判定 P0–P4

「どれだけ厳格に、誰が回すか」をリスクで機械的に決めます。P1〜P4はorch由来、P0は私たちの追加です。

パターン意味誰が設計・検収誰が実装典型
P0
拡張		探索性が高い（答えが未知）天才が直営天才が直営原因不明のデバッグ／設計と実装が不可分な新規
P1高リスク（金銭・個人情報・不可逆）天才（蒸留入力・短出力）割安モデル決済・認証・本番DB移行
P2標準指令塔が指示書＋検収割安モデル機能追加・中規模リファクタ
P3低リスク自走（詰まれば1問1答）割安モデル小改修・データ整形
P4定型・機械的セルフチェック可最安モデル一括置換・雛形生成

P0を足した理由

orchの物差しはリスク一軸。だが「危険ではないが探索的」な作業（未知のバグ調査など）はP3に落ち、天才がほぼ関与しないまま進む。要約から導けない問題を要約して渡すと答えごと失われる。P0はこの穴を塞ぐ脱出口で、根拠はorch自身の論理（「直読圏では委譲は固定費で負ける」）の裏返し＝「直営圏では蒸留は情報損失で負ける」。

04モデルの役割分担

「指令塔」は役割であってモデル名ではありません。入口が割安モデル（例：Opus）でも、最も判断が重い短い成果物だけを上位モデル（Fable）へサブエージェントとして外注できます。

3つの層

  • 指令塔（Conductor） — WP分解・採点・委譲の交通整理・検収。入口モデルが担う。
  • 判断オラクル（天才／Fable） — P1の設計書・迷い域の裁定・P1の検収最終判定・P0の直営。短く・判断が重い出力に限定。入力は蒸留を原則（P0のみ例外）。
  • 実行部隊（割安モデル／Opus・Sonnet） — 実装・調査・定型作業という「量のある仕事」。並列で回す。

なぜ安く済むか

コストは出力より入力トークンが支配する。天才への入力を「蒸留した調査報告のみ」に絞り、出力も設計書8–12k・裁定3行に制限することで、高い判断を少額で買える。実行部隊の大量トークンは割安モデルが引き受ける。

05実プロジェクトでの動き

あるEC監査プロジェクト（初適用例）での回り方を、機密数値を伏せて示します。1つのゴールが複数WPに割れ、それぞれ別パターンで処理されます。

WP（作業単位）判定の理由パターン回し方
停止していた顧客向けメール送信の復旧外部送信・不可逆（誤送信は取り消せない）P1原因調査は割安モデル→対策設計と「送信対象の限定」判断は上位モデル→実行前に承認ゲート
未設定の商品説明メタを一括生成定型・低リスク（下書き→人がレビュー）P4最安モデルで一括ドラフト→人が承認して反映
売上実績×在庫の優先リスト生成低リスク・既知データの集計P3割安モデルが自走で表を生成
原因不明の「特定流入だけ転換ゼロ」の調査答えが未知・全体文脈が要るP0天才が直営で仮説→検証（要約に耐えないため）

同じゴールでも、WPごとにコストと厳格さが最適化される。危険な1件だけ天才を厚く使い、量のある作業は割安に流す——これが「85点を安く買う」の実体です。

06他のSkillsとの連携

/orch は横断プロセス（メタ・スキル）です。ドメインを持つ他スキルを置き換えず、その上に「判定・委譲・検収」の型をかぶせます。指令塔が指揮者、他スキルが楽器という関係です。

典型的な組み合わせ

WPの種類orchの判定実作業で呼ぶスキル
ブランド文面・営業コンテンツ多くはP2/P3brand-voice（生成）＋ quality-assurance（検収）
課金・サブスク導入金銭+2 → ほぼP1stripe-provision / stripe-payment-link-kit
DBスキーマ変更・移行不可逆+2 → P1Supabase系ツール（検収でロールバック確認）
納品物（資料・表・PDF）P3/P4docx / pptx / xlsx / pdf
実装の検証検収3点の道具として/verify · /code-review · /security-review
調査・リサーチ常に安いモデルへdeep-research / Explore エージェント

連携の3原則

  1. orchは指揮、スキルは演奏。 orchが「どのWPを・どのパターンで・誰に」を決め、実作業は各ドメインスキルが担う。
  2. 検収は必ずorch側の3点セットで締める。 ドメインスキルの出力も、/verify 等を道具に使って指令塔が独立に検品する。
  3. プロジェクトのCLAUDE.mdが最優先。 独自のエージェント運用やセキュリティ規約があれば、orchはその内側で「WPの判定・委譲・検収」としてだけ働き、既存の型を上書きしない。

要するに

orchは「何をどの厳格さで進め、誰が検品するか」の交通整理。能力そのものは他スキルが供給する。だから既存のスキル資産を捨てずに、その上へ規律だけを足せる。

07F1/F2 と P0 の技術的意味

F1 / F2 — 天才の"入手可能性"で運転規則が切り替わる

  • F1（天才が枠内で安い）：Fableを指令塔に厚く立て、監査・型・ルールブックという恒久資産を作り置く。将来割安モデルが賢く動くための投資フェーズ。
  • F2（天才が従量／枠が枯渇）：既定は割安モデルの自走。Fable投入はP1の設計・裁定・検収とP0直営に限定し、必ず見積り→承認を通す。勝手な従量発生をゼロにする。
  • 切替はユーザー宣言。現フェーズの正はスポークの STATUS.md 冒頭に置く（規則本体には現在地を書かない＝陳腐化防止）。

プロンプトキャッシュと「降板」の無効化

orchには、従量の指令塔を守るためのキャッシュ規律や「降板プロトコル」があります。これらは指令塔が従量モデルである前提の装置。私たちの既定構成（入口＝定額モデル）では空回りするため明示的に無効化し、代わりに「天才を呼ぶ時は必ず承認ゲート」を鉄則に置き換えました。

入力が支配する、という事実

対話が伸びるほど、1ターンごとに会話全量の再読コストが積み上がる（出力は小さくても入力が大きい）。だからF2では、重い会話文脈を従量モデルに何度も読ませない——必要な判断だけを蒸留してサブエージェントに渡す設計になっている。

08導入手順

仲間が自分の環境に入れる場合の最小手順です（~ はホーム。Windowsは各自のパスに読み替え）。

# 1. ハブをclone（読み取り専用・常設）
git clone https://github.com/akiratsukakoshi/fable_orchestra ~/fable_orchestra

# 2. スキルを共通置き場へコピー（全プロジェクトで /orch 有効）
mkdir -p ~/.claude/skills
cp -r ~/fable_orchestra/skills/orch ~/.claude/skills/

# 3. SKILL.md 冒頭の「ハブのパス」をclone先に書き換える
#    （clone先を既定から変えた場合のみ）

各プロジェクトでは、初回に /orch を起動するとスポーク（docs/orchestration/STATUS.md ほか）が作られます。私たちの拡張（F1/F2・P0・承認ゲート）を効かせるには、プロジェクトの CLAUDE.md に上書き規約を10行ほど置きます。

更新の作法

ハブは git pull で追従。スキルを再コピーしたら、SKILL.mdのハブパス書き換えだけ再適用する（本体へのその他の編集はしない＝次回pullで消えるため）。カスタムは常にCLAUDE.md側へ。
